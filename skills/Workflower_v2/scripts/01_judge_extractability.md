# 阶段一：判断可提取性

**目标**：判断该聚类是否包含值得提取的有效工作流。

---

## 输入

- `selected_chains.json` - 包含多篇论文的推理链

---

## 操作步骤

### 第一步：加载并解析推理链

读取 `selected_chains.json`，提取：
- 推理链总数
- 涉及的论文 ID
- 每条链的高层结构（提到了哪些步骤）

### 第二步：识别公共模式

对推理链按相似性分组：
- 哪些操作在多条链中出现？
- 典型的步骤顺序是什么？
- 是否存在变体，还是只有一种主导模式？

**量化检查**：统计有多少条链共享相同的核心步骤序列。

### 第三步：应用工作流定义

对最常见的模式，逐一验证以下 5 项标准：

#### 1. 问题导向性
- 该工作流解决什么类型的问题？
- 问题描述是否清晰具体？（如"表征超导体磁阻特性"而非"研究材料"）

#### 2. 程序性
- 是否有清晰的步骤序列？（如 A → B → C）
- 能否按顺序列出所有步骤？

#### 3. 步骤依赖性
- 步骤 B 是否需要步骤 A 的输出？
- 调换顺序是否会破坏逻辑？

#### 4. 可泛化性
- 是否可应用于不同材料/样品？
- 还是仅适用于某篇论文的特殊设置？

#### 5. 清晰的输入输出
- 每个步骤的输入是什么？
- 每个步骤产生什么输出？

### 第四步：统计支撑数量

- 有多少条链（≥10？）遵循该模式？
- 若存在多个工作流，哪个最核心/最频繁？

---

## 决策逻辑

```
若 支撑链数 >= 10 且 全部5项标准满足：
    决策 = "可提取"
    理由 = "发现 {数量} 条链支撑工作流：{简要描述}"
否则 若 支撑链数 < 10：
    决策 = "不可提取"
    理由 = "支撑不足（{数量} < 10 条链）"
否则 若 未满足全部标准：
    决策 = "不可提取"
    理由 = "模式不满足标准：{具体哪项}"
```

---

## 输出格式

```json
{
  "decision": "可提取" | "不可提取",
  "reason": "...",
  "support_count": 15,
  "workflow_summary": "核心工作流的简要描述（包含解决的问题类型）",
  "criteria_check": {
    "problem_oriented": true,
    "procedural": true,
    "step_dependent": true,
    "generalizable": true,
    "clear_io": true
  }
}
```

---

## 常见误判

- **问题不明确**："研究某类对象" → 问题过于宽泛，不具问题导向性
- **过于具体**："在特定条件下处理特定样本" → 不可泛化
- **过于模糊**："分析数据" → 不具程序性
- **无依赖关系**："操作 A，操作 B，操作 C"（并行而非顺序）→ 不满足步骤依赖性
- **输入输出不清**："做分析"（输入是什么？输出是什么？）→ 不满足清晰 I/O

---
